# 從原始資料到營收的完整產品——六大支柱

將 65 家公司與 6 個框架的洞見，整理成你產品的分層結構。

---

## 產品管線（六層）

你的產品是一條六層管線，每一層各有對應洞見：

**① Data OS** → **② 處理引擎** → **③ 輸出層** → **④ 使用者** → **⑤ 商業模式** → **⑥ 護城河**

---

## 1. Data OS——取得並掌握原材料

*去把資料拿來、結構化、版本化、真正「擁有」它。*

- **資料留在來源，你建索引。** 不要整站複製政府網站。建中央索引，讓你能像單一系統一樣跨站搜尋。 *來源：Eos AI*
- **爬取 → 解析 → 正規化 → 版本化。** 結構偵測 → 表格抽取 → 實體抽取 → 交叉引用消解 → 司法管轄正規化 → 品質評分。這約佔八成工作量。 *來源：Fintool*
- **每個資料點需要：source_url、last_verified、confidence、effective_date、version_history。** 不像寫程式上下文，合規資料不能「大致正確」。 *來源：Fintool + Vulcan*
- **「記憶不是 RAG。」** 要跨時間追蹤事實；處理矛盾（舊規 vs 新規）；自動遺忘已失效法規。時間維度是核心。 *來源：Supermemory*
- **開源 schema，資料自己握。** GitHub 倉庫裡用 Markdown + YAML frontmatter；人人可貢獻，代理人經 CLI 或 API 取用。Schema＝標準＝鎖定效應。 *來源：Andrew Ng Context Hub*
- **政府來源超過 14.5 萬個。** Abstract 從中索引了 700 萬筆監管紀錄。先從 1 條法規、10 個司法管轄做起——不要一開始就 14.5 萬來源。 *來源：Abstract*

---

## 2. 處理引擎——把原始資料轉成可用知識

*把雜亂法規轉成結構化上下文的「轉譯層」。*

- **你是轉譯層，不是替代品。** 坐在原始來源與所有應用之間。不要取代政府網站、律師或合規人員——要在彼此之間**轉譯**。 *來源：Eos AI*
- **知識圖譜，不是文件庫。** 義務、要求、罰則、門檻、關係、機會——帶本體意識的邊；知識會更新、合併、矛盾、推論，從不只是 append。 *來源：Supermemory + Vulcan*
- **同一條管線產出三種格式。** Markdown 敘事（人讀）、JSON 後設資料（可搜、代理人讀）、結構化表格（門檻與截止日，兩者皆宜）。 *來源：Fintool*
- **在上下文之上加「技能層」。** 一有結構化資料，就加上「合規技能」——用 Markdown 寫的 SOP，告訴代理人**如何**做合規任務；非工程師可寫、無需部署。 *來源：Fintool*
- **變更偵測＝殺手級功能。** 不只「現在規則是什麼」，而是「改了什麼、何時改、影響誰」。主動通知勝過被動查詢。 *來源：Payna + Regology*
- **跨司法管轄的身分／概念對齊。** 同一概念在不同州叫法不同——例如把加州的 "surplus lines tax" 對到紐約的 "independently procured insurance tax"。 *來源：Eos AI*

## 3. 輸出層——如何交付價值

*MCP、API、技能、文件、Autopilot 服務。*

- **MCP Server 作為代理人的主要介面。** 工具例如：query_regulation、monitor_changes、find_opportunities、log_decision。透過 Claude Code、Cursor 等 MCP 外掛零摩擦採用。 *來源：Supermemory + 21st.dev*
- **Autopilot 服務作為人類的主要介面。** 不要只賣 API——**把合規工作做完**。客戶為**結果**付費（「我的申報做完了」），不是為基礎設施（「我有 API 權限」）。 *來源：Bek／Sequoia + Payna*
- **商業論證文件。** 自動產出合規報告、稽核包、監管申報文件；讓合規主管能交給稽核方一份成品。 *來源：Salesgraph*
- **稽核軌跡就是產品。** 每個合規判斷都要有「為何」——依據哪條法規、哪個門檻、推理為何；草稿本記錄每次工具呼叫與推理步驟。 *來源：Entire／Checkpoint + Fintool*
- **搭在前沿代理人之上，而非與之對抗。** 不要自研整套代理人框架；以 Claude Code SDK 或 21st Agents SDK 為底，專注合規專屬上下文、技能與工具。 *來源：21st.dev*

---

## 4. 使用者——誰用、怎麼用

*代理人、人類、混合流程。*

- **首要客戶＝其他公司的 AI 代理人（2A）。** 例如 Panta 的代理人查你 API 問 surplus lines 規則；Fenrock 查制裁邏輯；Parcha 查 KYC 門檻。你服務的是**代理人**，不是終端人類。 *來源：Entire + Eos AI*
- **次要客戶＝合規主管（Autopilot）。** 在做 API 的同時，直接代做合規工作創收；人類客戶驗證你的資料品質。 *來源：Bek／Sequoia*
- **上下文跟著使用者走。** 教會一個 AI 的合規知識，應能在各工具間攜帶，而不是鎖在單一平台。 *來源：Supermemory*
- **賦能內部擁護者（Champion enablement）。** 如同 Salesgraph 給業務文件去說服內部買方——你也要給合規主管材料，去說服領導投資合規。 *來源：Salesgraph*
- **律所是通路，不只是競品。** 服務管理多家客戶合規的律所；他們既是分發渠道，也是部分競爭者。 *來源：Payna*

## 5. 商業模式——如何賺錢

*從營收到護城河的順序：Autopilot → API → 平台。*

- **賣工作成果，不賣工具。** 下一個兆美元級公司可能是「偽裝成服務公司的軟體公司」——取代可計費工時，而不是只搶軟體預算。 *來源：Bek／Sequoia*
- **供應商替換，不是組織重組。** 從**已經外包**的任務切入：預算已在、買方習慣買「結果」，不必改客戶組織圖。 *來源：Bek／Sequoia*
- **免費層 → 用量計價 → 企業方案。** 例如：免費（1 個司法管轄）→ 開發者 $50/月 → 新創 $200/月 → 企業客製。Supermemory：Pro $29/月 → Scale $399/月 → 企業客製。 *來源：Supermemory + WorkOS*
- **混合營收：Autopilot（高毛利）+ API（經常性）。** Autopilot 建立信任並產生資料；API 把資料資產規模化變現；兩者互相餵養。 *來源：Checkr + Column*
- **零外部融資也可能。** Column 零外部資金做到約 2 億美元營收。若能很快靠 Autopilot 做出營收，未必需要創投。 *來源：Column*

## 6. 護城河——長期可防禦之處

*資料飛輪、schema 鎖定、以成果建立信任。*

- **資料飛輪。** 代理人查詢 → 驗證結果 → 知識變好 → 更多代理人。每次查詢讓資料更好；每次 Autopilot 專案增加可重用的結構化資料。 *來源：a16z 框架*
- **Schema 鎖定。** 若你的 JSON schema 成為法規結構化的業界標準，轉換成本極高；開源 schema 可加速採用。 *來源：Column + Andrew Ng*
- **法規變更歷史。** 鮮少有人系統性追蹤「法規**何時**改、**舊版**長什麼樣」——時間維資料是你的獨特資產。 *來源：Supermemory 時間追蹤*
- **以成果建立信任，不以頭銜。** 「這套資料處理 200 份政策零錯誤」勝過「有律師看過」。Norm AI 用 35 名律師；你用**結果數據**。 *來源：Eos AI + Norm AI*
- **底層不可被繞過。** 不要當聰明 LLM 能跳過的中間層（如 Plaid／Norm AI 類比）；要當底層（如 Column）——任何 LLM 仍需要的原始結構化資料。 *來源：Column*
- **模型會吃掉你的腳手架。** 設計上要預期淘汰：多用 Markdown、少綁死程式。留下的是：資料、客戶關係、領域知識；會消失的是：下一版模型原生就能做的事。 *來源：Fintool + 21st.dev*

## 你原先摘要缺了什麼

完整圖景與你已捕捉內容之間，有四處缺口。

### 缺口 1：信任機制

你寫了產品架構（Data OS → 上下文 → API），但漏了**信任層**。合規領域沒有信任產品跑不起來——而信任要靠 **Autopilot 的成果**建立，不能只靠資料品質工程。這是 Eos AI 的洞見：臨床級準確需要**成果驗證**，不只工程卓越。

### 缺口 2：時間智能

你提到「處理資料」，但沒有單獨拉出**時間維度**。Supermemory 的關鍵是：記憶 ≠ 檢索。追蹤規則**何時**變、消解新舊版本矛盾、自動淘汰過時法規——這與「只結構化當前規則」是根本不同的能力。

### 缺口 3：商業模式順序

你列了「合規利基、autopilot、complyance」等衍生利基，但沒寫清**順序邏輯**——為何 **Autopilot 要先**（信任 + 營收 + 產生資料），**API 在後**（須先有資料與可信度才成立）。順序比元件清單更重要。

### 缺口 4：內部擁護者賦能

你聚焦服務代理人与組織，但漏了 Salesgraph 式洞見：產品也要服務**內部倡議者**——需要向領導論證合規投資的合規主管。自動產生的稽核包與合規報告，就是在服務這個角色。

## 反模式——不要做什麼（來自 65 家公司的教訓）

### 不要打造

- 第一天就做全球合規平台（Regology：8 年、800 萬美元級投入參考）。
- 自己的代理人框架（21st.dev 曾為此付出代價）。
- 財星百大等級的企業直銷（Norm AI 的戰場，約 1.47 億美元先發優勢）。

### 不要混淆

- **研究**與**進展**（你：65 家公司、0 條結構化法規）。
- 漂亮的 README 與 PMF（GENESIS v10.1：README 很美、0 客戶）。
- 橫向願景與垂直執行（從 1 條法規開始，不是「所有合規」）。

### 不要假設

- 客戶**已經**強烈感受到痛點（需求缺口是頭號致命風險）。
- **基準測試**在高風險領域等於信任（Supermemory 可以靠榜單領先；你需要**成果證明**）。
- 沒有 **Autopilot 先行**，**API 先行**也能冷啟動成功（冷啟動會殺死純 API 玩法）。





## 你的總結差什麼

你抓到的核心骨架是對的——「Data OS → 抓取處理 → 轉換成 context → 提供給 agent/human/API」。這確實是產品的前三層。但你的總結缺了三個關鍵層。

**第一個缺的是「怎麼賺錢」和「先後順序」。**

你提到了 Autopilot 和 compliance niche，但沒有把它們放進時間線。65 家公司反覆教你的最大一課是：**順序比組件重要。** Autopilot 必須先做，不是因為它是最好的商業模式，而是因為它同時解決三個問題：產生收入（生存）、累積結構化數據（燃料）、建立信任（門票）。API 是第二步，因為沒有數據和信任，沒人會 integrate 你的 API。這就是 Checkr、Norm AI、[21st.dev](http://21st.dev) 都走過的路——先做一件很具體的事，做出成績後再抽象化。

**第二個缺的是信任機制。**

合規跟其他 context layer 的根本差異就在這裡。Salesgraph 的 context 錯了，最多是銷售代表說錯一個數字。Supermemory 的記憶錯了，最多是 AI 推薦了錯誤的偏好。但你的合規 context 錯了，客戶可能被罰款、被吊銷牌照、甚至面臨刑事責任。所以你不能像 Supermemory 一樣靠 benchmark 領先來建立信任，你需要像 Eos AI 一樣靠臨床結果（＝合規結果）來建立信任。

**第三個缺的是時間維度。**

你說「抓取並處理整理資料」，但沒有區分「靜態結構化」和「動態追蹤」。Supermemory 教你的最重要一課是：memory ≠ retrieval。你的產品不只是「加州 surplus lines 稅率是 3%」——而是「加州 surplus lines 稅率在 2024 年 1 月從 3% 改為 3.5%，影響了所有未繳稅的保單，新規適用於 2024 年 3 月 1 日後的交易」。追蹤變化歷史、解決新舊規則矛盾、自動過期失效法規——這是你的競爭對手沒做的事，也是你最深的護城河。

## 一句話版本

如果要把 65 家公司的所有 insight 壓縮成一句話：

**「把散落在各處的法規數據結構化成有時間維度的知識圖譜，先通過 Autopilot 服務賺取信任和收入，再把累積的數據資產通過 MCP/API 賣給所有需要合規知識的 AI agent。」**

這句話裡每個詞都對應一個具體的 insight 來源——「散落各處」是 Eos AI 的翻譯層，「時間維度」是 Supermemory 的 memory ≠ RAG，「Autopilot 先」是 Bek/Sequoia 的 playbook，「信任」是 Norm AI 的反面教材，「MCP/API」是 [21st.dev](http://21st.dev) 和 Supermemory 的分發策略，「所有 AI agent」是 Entire 的 2A 概念。
