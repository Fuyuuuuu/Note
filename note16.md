這個問題是對的。如果不能壟斷，就是在跟所有人打價格戰。

## 先看合規 AI 裡誰已經有壟斷性

**Checkr** — 壟斷了「背景調查 API」這個品類。$800M 收入。Uber、Lyft、Airbnb 都用。壟斷來源不是技術，是**網絡效應**：查過的人越多 → 數據越完整 → 結果越快越準 → 更多公司用 → 查過的人更多。

**Plaid** — 壟斷了「銀行帳戶連接 API」。壟斷來源是**連接數量**：接了 12,000+ 金融機構。任何新進者要從零開始接這些銀行，要花數年。

**Norm AI** — 正在嘗試壟斷「法規→AI 可執行格式」。壟斷來源是**人力密集的知識工程**：35+ 律師手動編碼法規成決策樹。但這恰好也是它的弱點——用人力建的護城河，可以被更好的技術繞過。

## 壟斷的五種來源

在合規 AI 領域，真正能建立壟斷的只有以下幾種：

**① 數據壟斷——你有別人拿不到的數據**

Checkr 的壟斷就是這個。每做一次背景調查，它的數據庫就多一筆記錄。十年後沒人追得上。

在合規領域：如果你能累積「法規的實際執行方式」的數據——不是法規文本（公開的），而是「這條法規在加州 Alameda County 實際怎麼被執行、罰了誰、罰了多少」——這是別人沒有的。法規文本人人都拿得到，法規的實際執行數據沒人有。

**② 網絡效應——用戶越多，產品越好**

Plaid 的壟斷。每多一個銀行連接，對所有 fintech 客戶都更有價值。

在合規領域：如果你的 API 被 100 家 AI 公司調用，每家的 agent 在使用過程中回報「這個法規解讀是對的」或「這個有誤」，你的知識圖譜就越來越準。第 101 家公司用你的 API 時，得到的是被 100 家公司的 agents 驗證過的知識。任何新進者從零開始，沒有這個驗證循環。

**③ 標準壟斷——你定義了行業標準**

Stripe 壟斷了「線上支付 API 的介面標準」。不是因為它的技術不可複製，而是因為所有開發者都學了 Stripe 的 API 格式。切換成本太高。

在合規領域：如果你定義了「合規知識的 schema」——法規怎麼被結構化、obligation 的格式是什麼、requirement 的表達方式是什麼——而且 50 家 AI 公司的 agents 都按你的 schema 寫 prompt，你就成了事實標準。即使有人做出更好的數據，他們也要兼容你的格式。

**④ 嵌入壟斷——切換成本高到不值得換**

Salesforce 的壟斷。不是因為它最好，而是因為整個公司的流程都建在上面，換掉的成本比忍受它更高。

在合規領域：如果 Panta 的 AI agent 每天調用你的 API 3,000 次來做保險合規判斷，所有的 prompt、workflow、edge case 處理都是按你的 API 回應格式設計的，換一個供應商意味著重寫整個 agent 的合規邏輯。沒人會為了省一點 API 費用去承擔這個風險。

**⑤ 知識圖譜壟斷——結構化的深度無法複製**

Google 的 Knowledge Graph。不是爬了多少網頁，而是實體之間的關係建得多深。

這是 Vulcan 在嘗試的。也是你最應該追求的。

## 你能壟斷什麼

坦白說，作為一個人、沒融資、在台灣，你現在不可能壟斷任何東西。但你可以**選擇一個能在未來產生壟斷性的方向，從現在就開始累積**。

我認為你最可行的壟斷路徑是 **② + ③ + ⑤ 的組合**：

**建一個特定領域的合規知識圖譜，定義它的 schema 標準，然後用 API 調用者的反饋形成網絡效應。**

具體來說：

**第一步（現在）：選一個碎片化的法規領域，手動建構知識圖譜。**

不是爬法規文本（Regology 已經做了），而是把法規結構化成「義務→要求→條件→罰則→例外→關聯法規」的圖譜。你定義 schema。

**第二步（3-6 個月）：開源 schema，讓人開始用。**

schema 本身是免費的。就像 JSON Schema 或 OpenAPI Spec。任何人都可以用你的格式來結構化法規。但你是定義者。

**第三步（6-12 個月）：賣填好的數據。**

schema 免費，數據收費。就像 OpenStreetMap 的 schema 免費，但 Mapbox 賣基於它的地圖服務。

**第四步（12+ 個月）：API 調用者的 agents 回報驗證數據，形成飛輪。**

每次 agent 用你的 API 做了一個合規決策並且結果正確/錯誤，這個信號回流到你的圖譜。圖譜越來越準。新進者沒有這個反饋循環。

## 最終壟斷狀態

**你壟斷的不是法規文本（公開的），不是 AI 技術（通用的），而是「法規知識被 AI 消費的標準格式 + 被實際使用驗證過的知識品質」。**

就像 Checkr 壟斷的不是「犯罪記錄」（公開的），而是「快速、準確、API-first 的犯罪記錄查詢，被十萬家公司驗證過的結果品質」。

這需要時間。但方向選對了，每一天的累積都在加深護城河。選錯了，做十年也沒有壟斷性。

## AI 時代的壟斷新公式

把 Thiel + YC 合在一起，AI 時代的壟斷公式是：

**壟斷 = 專有數據 × 反饋飛輪 × 標準鎖定 × 小到可以主宰的起點**

對應到你：

**① 專有數據：不是法規文本，而是「法規的實際執行數據」**

**③ 標準鎖定：定義 schema，而不只是填充數據**

你不只賣「法規數據」。你定義「法規數據的格式」——obligation、requirement、condition、penalty、exception、jurisdiction、effective_date 的標準結構。開源 schema，讓所有合規 AI 公司按你的格式建 prompt。一旦 50 家公司的 agent 都按你的 schema 設計，你就是事實標準。

**④ Thiel 的小市場起點：一個管轄區 × 一個法規類型 × 一個客戶**

Thiel 堅持每個新創公司都應該從小市場開始，因為主宰利基市場比主宰大市場更容易。 PayPal 從 eBay PowerSellers 開始，不是從「全球支付」開始。

你的 PayPal eBay 時刻：**台灣金管會的某一類特定法規 × 結構化成 JSON schema × 給一家台灣 fintech 的 AI agent 用。**

## 合規 Context API 是幾性需求？

## API 在現實中的摩擦

就算需求變強了，「合規 Context API」這個產品型態在現實中有真實的摩擦：

**摩擦 1：信任門檻極高**

合規決策是高風險決策。做錯了 → 罰款、吊銷牌照、刑事責任。沒有任何一家公司會因為你的 API 返回了一個 JSON 就相信那是正確的法規解讀。他們會問：「你的數據來源是什麼？誰驗證的？有律師背書嗎？出錯了誰負責？」

Norm AI 解決這個問題的方式是：雇 35 個律師。你沒有 35 個律師。

**摩擦 2：整合成本**

一家 Autopilot 公司要接你的 API，意味著他們要改自己的 agent 架構。從「合規知識寫在 prompt 裡」改成「每次決策前先調用外部 API」。這是架構級的改動，不是加一行 code 的事。早期新創沒有動力做這個改動——因為現有方式「夠用了」。

**摩擦 3：Cold start**

你的 API 第一天上線時，覆蓋的法規範圍一定很窄。可能只有一個州的一個法規類型。對 Panta 來說，如果你只有加州 surplus lines 但沒有德州的，他們不會為了一個州去接你的 API——他們寧可自己查。要到你覆蓋夠廣的時候才有吸引力，但覆蓋夠廣需要大量時間和成本。雞生蛋的問題。

**摩擦 4：抽象 vs 具體**

「合規 Context API」是一個抽象的基礎設施概念。但客戶買的從來不是抽象概念，是具體的結果。Checkr 不是賣「身份驗證 API」，是賣「3 分鐘內告訴你這個人能不能雇」。你的 API 回答的具體問題是什麼？如果答案是「任何法規問題」，那就太抽象了，沒有人知道怎麼用。

## 那什麼是第一性的？

回到根本。第一性需求只有兩個：

**「幫我完成這件合規任務」（Autopilot）**
→ Panta：幫我綁定這張保單
→ Payna：幫我拿到這個執照
→ Noetic：幫我通過這個認證

**「幫我不出事」（合規保障）**
→ 確保我的 AI agent 沒有違規
→ 確保我擴張到新市場時不會踩雷

API 是服務這兩個第一性需求的手段之一，但不是唯一手段，也不是最直接的手段。

## 所以你應該怎麼想

**不要從產品型態出發，從客戶的第一性需求出發。**

錯誤的思考順序：「我要建合規 Context API」→「誰會用？」

正確的思考順序：「誰有什麼具體的合規痛點？」→「最直接的解決方式是什麼？」→ 那個解決方式可能是 API，也可能是 Autopilot 服務，也可能是一份結構化的文件，也可能只是一通電話。

**這就是為什麼「下場做 Autopilot」比「先建 API」更好的根本原因。**

做 Autopilot 時你在解決第一性需求：幫客戶完成一件具體的事。在這個過程中，你自然會建構合規知識庫。如果這個知識庫對別人也有用，它自然會演化成 API。

但 API 是結果，不是起點。

如果你從 API 開始，你在賣的是一個你自己都沒用過的抽象工具，給一群還沒意識到自己需要它的客戶。這就是為什麼它會在現實中產生摩擦。

## The Mom Test 的核心 insight

這本書的重點不是「怎麼做用戶訪談」。核心 insight 是：**人們會對你的想法撒謊（出於禮貌），但不會對自己的行為撒謊。**

三條規則：

**① 問他們的生活，不要問你的想法。**
錯：「你覺得合規 API 有用嗎？」（他們會說有用，因為不想傷你的心）
對：「你上次碰到合規問題時，你怎麼解決的？花了多少時間？」

**② 問過去的具體行為，不要問未來的意願。**
錯：「如果有這個工具你會用嗎？」（所有人都說會）
對：「你上次為了搞清楚德州的 surplus lines 規則花了多久？最後怎麼處理的？」

**③ 如果他們沒有花時間或金錢試圖解決這個問題，這個問題就不夠痛。**

## 你用得上的地方

**不是用在「驗證你的 API 想法」。是用在你下場做 Autopilot 的過程中。**

當你找到那一個人、準備幫他做一件具體的合規任務時，Mom Test 幫你判斷：**這件事對他來說是真痛還是假痛。**

比如你認識一家台灣 fintech，你覺得他們可能需要幫忙申請某個執照。Mom Test 的做法不是問「你需不需要幫忙」，而是：

「你們現在的執照申請是誰在處理？」
「上次申請花了多久？」
「過程中最卡的地方是什麼？」
「你們為這個付了多少錢給律師/顧問？」
「有沒有因為執照問題延遲過產品上線？」

如果答案是「我們花了三個月、付了律師 50 萬台幣、產品因此延遲上線兩個月」——這是真痛。

如果答案是「我們的法務自己處理，大概花了兩週，還好」——這不夠痛，換一個問題。

## 最關鍵的 Mom Test insight 對你的適用

書裡有一個概念最適合你現在的狀態：

**「如果你在對話結束後沒有得到一個承諾（commitment），這次對話就是浪費時間。」**

承諾不是「聽起來不錯」。承諾是具體的行動：

弱承諾：「好有趣，保持聯繫。」（沒用）
中等承諾：「你做好了發給我看看。」（稍微有用）
強承諾：「我可以把我們現在的執照申請資料給你，你幫我試試。」（真的有用）
最強承諾：「我們願意付你 X 元來做這件事。」（就是這個）

**Goldbridge 路線：** 你選一個垂直（比如台灣 fintech 合規），深度理解這個垂直的每一個痛點，幫客戶從頭到尾搞定合規——申請執照、追蹤法規變更、準備監管審查、處理違規。你就是他們的合規部門。

## 各自的優劣

**Column 路線的優勢：**

規模效應。一套 API 服務所有垂直。不需要理解每個行業的具體痛點。

客戶多元化。不依賴單一行業。保險不好做了，還有金融執照、還有硬體認證。

護城河累積。每多一個垂直的法規數據，API 對所有客戶都更有價值。

**Column 路線的劣勢：**

Cold start 極難。你的 API 第一天只有一個垂直的數據，對大多數潛在客戶沒用。

信任門檻高。合規數據出錯的代價太大。沒有人會信任一個沒有經過實戰驗證的 API。

收入慢。API 按次計費，需要大量客戶才能有可觀收入。

**Goldbridge 路線的優勢：**

立刻有收入。替代定價——客戶本來付律師的錢付給你。第一天就可以收費。

信任容易建立。你親手幫客戶做合規，客戶看到結果，信任你的能力。

深度理解產生洞察。你在第一線做合規，會發現 API 設計者永遠發現不了的邊緣案例。

**Goldbridge 路線的劣勢：**

規模受限。每個垂直的合規知識不同，進入新垂直要重新學。

人力密集。至少一開始是手動做，不是 API 自動回答。

天花板較低。除非你能把手動流程自動化成平台。

## Goldbridge 的真正洞察

Goldbridge 最聰明的地方不是「做房地產 fintech」。是這個：

**Goldbridge 坐在銀行+交易層，所以它擁有比任何純軟體公司更乾淨、更豐富的數據。**

因為 Goldbridge 處理房東的錢（不只是看數據），它知道每一筆交易的真實情況——水電費多少、租金什麼時候收到、維修花了多少、房產稅交了沒有。這些數據讓它的 AI 建議比任何外部工具都準。

**翻譯到你的情境：如果你親手幫客戶做合規（Goldbridge 模式），你就擁有「合規決策的實際執行數據」——哪些法規解讀是對的、哪些邊緣案例容易出錯、監管機構實際上怎麼執法。這些數據是只做 API 的人永遠拿不到的。**

## 最深的洞察：Column 和 Goldbridge 不是對立的

Column 提供銀行基礎設施。Goldbridge 是 Column 的（潛在）客戶——用 Column 的管道來服務房東。

在你的世界裡：

**你可以同時是 Column 和 Goldbridge。**

自己做一個垂直的 Autopilot 合規服務（Goldbridge 模式），在這個過程中建構的合規知識變成 API 給其他 AI 公司用（Column 模式）。

Goldbridge 是你的前台——面向客戶、產生收入、獲取實戰數據。 Column 是你的後台——結構化知識、API 化、服務更廣的生態系。

這就是為什麼「三個角色同時進行」是最好的策略：

你的 Goldbridge（垂直 Autopilot 服務）→ 產生收入 + 實戰數據 你的 Column（合規知識 API）→ 服務其他 AI 公司 + 建護城河 你的佈道 → 讓兩者都被看見

**Goldbridge 沒有 Column 就沒有銀行基礎設施。Column 沒有 Goldbridge 這樣的客戶就沒有收入。**

**你的 Autopilot 服務沒有合規知識層就做不好。你的合規知識層沒有 Autopilot 服務的實戰驗證就沒人信。**

**兩者需要同時存在。不是選擇題。**

## Hockey 為什麼離開 Plaid 去做 Column

這不是公開場合大談的故事，但從他的決策可以逆向推導：

**Plaid 的結構性天花板：你翻譯得再好，原文爛就是爛。**

Plaid 連接了 12,000 家銀行。但這 12,000 家銀行的核心系統是幾十年前的。API 不穩定、數據格式混亂、即時性差。Plaid 花了巨大的工程力量去「翻譯」這些爛系統，讓它們看起來統一和乾淨。

但 Plaid 永遠無法讓銀行的系統變得更好。它只能讓爛系統「看起來不那麼爛」。

**Hockey 的洞察：與其花一輩子翻譯爛的東西，不如從零建一個好的東西。**

## 七個維度的深層對比

**① 控制權**

Plaid 不控制銀行的 API。銀行改了介面，Plaid 就要重新適配。銀行擋了 Plaid 的存取，Plaid 就斷了（事實上多家大銀行曾封鎖 Plaid）。

Column 控制一切。因為它就是銀行。沒有人可以封鎖你自己。

**啟發：做中間層的人永遠在別人的許可下運作。做底層的人自己給自己許可。**

**② 價值捕獲**

Plaid 每次連接收費。但 Plaid 的連接只是一個動作——連完了就完了。Plaid 不參與後續的資金流動、不賺交易的利差、不持有存款。

Column 賺交易費、利息收入、卡片交換費、貸款利差。每一筆通過 Column 的資金，Column 都在持續賺錢。資金在 Column 裡停留越久，Column 賺越多。

**啟發：中間層只在「連接的瞬間」賺錢。底層在「資產存續的整個生命週期」賺錢。**

**③ 客戶關係**

Plaid 的客戶用 Plaid 連接銀行帳戶，連完後就不太需要 Plaid 了（直到下次連接）。Plaid 是「用完即走」的工具。

Column 的客戶把整個業務建在 Column 上。Brex 的每一筆交易、Mercury 的每一個帳戶、Ramp 的每一張卡片都跑在 Column 上。Column 是「永遠在線」的基礎設施。

**啟發：工具可以被替換。基礎設施不能被替換。**

**④ 護城河的性質**

Plaid 的護城河是**連接數量**——12,000 家銀行的整合。這是真實的護城河，新進者要花數年才能追上。但這個護城河有一個弱點：如果銀行開始提供標準化的 API（比如透過 Open Banking 法規），Plaid 的翻譯價值就降低了。

Column 的護城河是**牌照 + 自建技術 + 直連聯準會**。銀行牌照不會因為法規變化而貶值。自建的核心系統不會因為標準化而被商品化。直連聯準會的能力全美只有少數機構有。

**啟發：基於「別人的系統很爛所以需要翻譯」的護城河，會隨著別人的系統改善而消失。基於「我擁有不可替代的資產」的護城河是永久的。**

**⑤ 商業模式韌性**

Plaid 面臨多重威脅：銀行自己開 API（Open Banking）、競爭對手（MX、Finicity 被 Mastercard 收購）、大客戶自己建連接。$13.4B 估值但面臨壓力。

Column 幾乎沒有直接競爭者。要競爭 Column，你需要：買一家銀行（$50M+）、從零建技術堆疊（3 年）、取得監管信任（多年）、簽下 Brex 級別的客戶。全部加起來，進入門檻是天文數字。

**啟發：中間層的進入門檻是「工程努力」（可以用錢和人砸出來）。底層的進入門檻是「時間 + 牌照 + 信任」（砸錢也加速不了）。**

**⑥ 規模經濟**

Plaid 每增加一家銀行連接，需要工程團隊去做適配。12,000 家銀行 = 12,000 種不同的整合要維護。規模越大，維護成本越高。是線性擴張。

Column 的技術堆疊是統一的。每增加一個客戶，邊際成本接近零。Brex 用的跟 Mercury 用的是同一套系統。是指數型擴張。

**啟發：翻譯層的成本隨覆蓋範圍線性增長。底層的成本是固定的，收入隨客戶量指數增長。**

**⑦ 對 AI 時代的適應性**

Plaid 在 AI 時代面臨一個問題：AI agent 需要的是「直接存取金融數據和操作」，不是「翻譯一層然後給我」。如果 Column 直接提供 AI-ready 的 API，AI agent 為什麼要繞道 Plaid？

Column 天然適合 AI 時代：API-first、開發者友好、即時、可程式化。AI agent 可以直接用 Column 的 API 開帳戶、轉帳、發卡。

**啟發：AI 時代偏好直接連接到底層，而不是通過翻譯層。中間層在 AI 時代面臨被跳過的風險。**

## 從所有 session 的 insight 合成你的產品

### Insight 1：從 Nia 學——MCP Server 是對的交付形態

Nia 不是一個網站、不是一個 dashboard、不是一個 SaaS 平台。它是一個 MCP server——直接嵌入 Cursor、Claude Code 等 agent 的工作流程裡。開發者不需要「離開 agent 去查東西」，agent 自己去查。

合規 AI agent 不需要「離開自己的流程去查法規」。你的 MCP server 嵌入它們的工作流程，agent 在做決策的瞬間自動查詢你的知識。

### Insight 2：從 Column 學——自己是 Ground Truth

Nia 索引的是別人的東西（GitHub repos、npm packages）。這意味著 Nia 的價值隨著被索引的東西改善而降低——如果 Cursor 自己直接讀 GitHub，Nia 的翻譯價值就下降。

你不能犯同樣的錯。你不是索引 Regology 的數據或爬政府網站然後轉發。你是**自己從法規原文結構化，成為來源本身**。

Column 直連聯準會，不經過中間商。你直連法規原文，不經過第三方數據庫。

### Insight 3：從 Fintool 學——Context 是產品，80% 的工作是解析和正規化

之前研究的 Fintool 框架說「context is the product」，而且 80% 的工作是解析和正規化。

法規文本是 PDF、HTML、掃描件、模糊的法律語言。把「加州保險法第 1763 條」變成 agent 可以直接消費的 JSON——這個解析和正規化的工作就是你的產品。不是很酷的 AI 功能，是苦功。但苦功就是護城河。

### Insight 4：從 Checkr 學——數據飛輪

Nia 每次被 agent 查詢，都能學到「什麼知識最常被需要」。你也一樣。

每次 Panta 的 agent 查你的 API，你知道保險領域最常需要哪些法規知識。每次 Fenrock 的 agent 查你的 API，你知道 AML 領域最常需要哪些跨境規則。

這些查詢模式本身就是數據資產。它告訴你下一步該結構化什麼法規。新進者沒有這個信號。

### Insight 5：從 Goldbridge 學——不只是防禦，更是進攻

大部分人想到合規就想到「避免被罰款」。但 Goldbridge 用法規知識幫客戶找到房產稅上訴機會、政府退稅。

你的 MCP server 不只回答「這個行為合不合規」（防禦），也回答「有什麼法規上的機會你不知道」（進攻）。比如「你的客戶可能符合這個州的小企業免稅條件」或「這個新法規草案如果通過，你的產品可以進入一個新市場」。

防禦型知識是 cost center（客戶不喜歡付錢避免壞事）。進攻型知識是 profit center（客戶喜歡付錢得到好事）。

### Insight 6：從 Cumbuca 學——最薄的一層

Cumbuca 沒有建「Cumbuca API」，而是建了最小的法規層。客戶直接對官方規範開發。

你的 MCP server 不要試圖做「合規決策引擎」（太厚、太有主見）。做最薄的知識層——回傳結構化的法規事實，讓 agent 自己做判斷。

Nia 不幫 Cursor 寫 code。Nia 告訴 Cursor「這個 package 的 API 長這樣」，Cursor 自己決定怎麼用。

你不幫 Panta 做合規判斷。你告訴 Panta「加州 surplus lines 的規則是這些」，Panta 的 agent 自己決定怎麼處理。

### Insight 7：從 Thiel 學——開源 schema，壟斷 data

Nia 的 GitHub 是開源的。任何人都能看到它怎麼運作。但 Nia 預索引了 3,000+ packages 的數據——這個數據量是新進者不能一夜複製的。

你的 schema（法規結構化格式）應該開源。任何人都能用你定義的 obligation/requirement/condition/penalty 格式。但你填好的結構化法規數據是付費的。

Schema 越多人用 → 你的格式成為標準 → 標準鎖定 → 壟斷。

### Insight 8：從 Bek/Sequoia 學——Model 會吃掉你的 scaffolding

Fintool 的框架警告：「設計時要預期你的架構會被 model 能力的提升淘汰。」

如果你的價值只是「幫 agent 讀 PDF」——明年 Claude 6 可能自己就能完美讀 PDF 了。你的價值要在 model 能力提升後仍然存在。

什麼不會被 model 吃掉？**持續更新的、被驗證過的、結構化的法規知識圖譜。** Model 可以更好地讀法規原文，但不能替代「追蹤全球法規的每日變更、驗證結構化數據的準確性、維護法規之間的關係圖譜」。

就像 model 可以更好地讀程式碼，但不能替代 Nia 持續監控 3,000+ packages 的更新。

## 合成後的產品定義

**名字先不管。產品是什麼：**

**一個 MCP server，為合規 AI agent 提供持續更新的、結構化的法規知識。**

**具體功能：**

**法規查詢。** Agent 問「加州 surplus lines 的稅率和申報要求」，你回傳結構化 JSON，包含具體數字、生效日期、來源引用、相關法規連結。

**法規監控。** 某個管轄區的法規更新了，你的系統自動偵測、重新結構化、推送給訂閱的 agent。Nia 持續監控 package 更新，你持續監控法規更新。

**法規關係圖譜。** 不只是單條法規，而是法規之間的關係——「這條州法受這條聯邦法約束」「這條法規被這個判例修改了」。來自 Vulcan 的知識圖譜概念。

**機會偵測。** 不只是「你需要遵守什麼」，也是「有什麼法規上的機會」。來自 Goldbridge 的洞察。

**審計追蹤。** 每次 agent 查詢你的 API，都有記錄。「這個合規決策是基於哪條法規、在什麼時間、用了什麼版本的數據」。來自 Fintool 的 scratchpad 審計概念。

**不做的事：**

不做合規判斷（太厚，讓 agent 自己判斷）。不做工作流程管理（那是 Complyance/Norm AI 做的）。不做終端用戶介面（你的客戶是 agent，不是人）。

## Layer 2：合規知識圖譜（你的核心資產）

這是整個產品最關鍵的一層。Nia 索引已經結構化的程式碼。你要結構化還沒被結構化的法規。這層的苦功就是你的護城河。

**你的 Schema（開源）長什麼樣：**

```json

  "requirements": [

      "rate": 0.03,

    },

    "fine_range": {"min": 5000, "max": 50000},

        for large commercial risks",

    }
```

**四個關鍵設計決策：**

**obligation vs requirement 分離。** Obligation 是「你必須做什麼」（抽象的義務）。Requirement 是「具體怎麼做」（數字、表格、流程）。分開是因為同一個 obligation 在不同州有不同的 requirements。這來自 Cardamon 的 regulation → obligations → requirements 框架。

**relationships 欄位。** 法規不是孤立的。「CA-INS-1763 受 CA-INS-1760 約束」「聯邦法 X 優先於州法 Y」。這來自 Vulcan 的知識圖譜概念。Nia 不需要這個（程式碼之間的關係很簡單），但法規之間的關係是理解合規的核心。

**opportunities 欄位。** 這是 Goldbridge 的洞察——法規不只有義務，也有機會。大部分合規工具不告訴你「你可以利用什麼」，只告訴你「你必須遵守什麼」。這個欄位讓你的產品從 cost center 變成 profit center。

**confidence + last_verified。** 這是解決信任問題的關鍵。每條數據都有可信度分數和最後驗證日期。agent 可以根據 confidence 決定是否需要人類複核。來自 Fintool 的 eval 概念。

## Layer 3：MCP Server（Nia 的模式）

這是你的交付層。Nia 用 MCP protocol 讓 Cursor 直接調用。你用同樣的 protocol 讓合規 AI agent 直接調用。

**四個 MCP Tools：**

**Tool 1：**`query_regulation`

Agent 問一個具體問題，你回傳結構化答案。

```
Input:  "What are the surplus lines tax 
         requirements in California?"
Output: JSON with obligation, requirements, 
        penalties, source_url

```

這是最基礎的功能。Nia 的 `nia_package_search` 等價物。

**Tool 2：**`monitor_changes`

Agent 訂閱某個管轄區或法規類型的變更通知。法規更新時自動推送。

```
Input:  subscribe("US/CA/insurance/surplus_lines")
Output: webhook when regulation changes

```

Nia 持續監控 package 更新。你持續監控法規更新。這是你跟「靜態法規資料庫」的核心差異——你是活的。

**Tool 3：**`find_opportunities`

這是你獨特的。其他合規工具不做這個。

```
Input:  company_profile (industry, size, 
        jurisdictions, current_licenses)
Output: list of tax breaks, exemptions, 
        incentive programs the company qualifies for

```

Goldbridge 幫房地產業主找到房產稅上訴機會。你幫 AI agent 的客戶找到法規裡的機會。

**Tool 4：**`log_decision`

每次 agent 用你的數據做了合規決策，記錄下來。「在 2026-03-23，Panta 的 agent 查詢了 CA surplus lines 稅率，基於 regulation CA-INS-1763 v2024-01-01，做出了 X 決策。」

這是 Fintool 的 scratchpad 概念。合規審計時，監管機構要看「你的 AI 為什麼做這個決定」。你的 audit trail 就是答案。

## Layer 4：AI Agent 客戶

你不面向終端用戶。你的客戶是 AI agent 公司。

**第一批目標客戶（從你研究的 64 家裡）：**

Panta（保險，需要 50 州 surplus lines 法規）、Fenrock（AML，需要跨境 AML 法規）、Noetic（硬體認證，需要 FCC/CE/UL 標準）、Eloquent AI（金融服務，需要跨國金融法規）。

但你不需要四個都服務。你只需要一個。

## 數據飛輪（Checkr 的模式）

**這是你長期壟斷的核心。**

每次 agent 查詢你的 API → 你知道哪些法規最常被查 → 你優先完善這些法規的深度 → 準確性提高 → 更多 agent 用你 → 更多查詢數據。

而且每次 agent 的合規決策結果回報（成功/失敗/被監管機構質疑），你的 confidence 分數就更準確。第 1000 次被驗證的法規數據，新進者不可能一夜複製。

## 最重要的：第一天做什麼

**不是建 MCP server。不是設計 schema。不是寫爬蟲。**

**是打開一個政府網站，讀一條法規，用你的手和腦把它變成上面那個 JSON 格式。**

一條。

如果你選跨境電商（之前建議的起步利基），第一條可能是：「FCC 對消費電子產品的 Declaration of Conformity 要求」。

如果你選保險（最有價值的利基），第一條可能是：「加州 surplus lines 的稅率和申報要求」。

如果你選台灣（最容易開始），第一條可能是：「台灣金管會對第三方支付業者的資本額要求」。

**做完這一條，你就有了：schema v0.1、第一塊結構化數據、可以發到 X 上的第一篇 build in public 內容、可以拿給一個人看的 demo。**

然後第二條。第三條。第十條。到第五十條的時候，你就有了一個 MCP server 的雛形值得讓 agent 試用。

**Nia 從索引一個 package 開始。你從結構化一條法規開始。**
