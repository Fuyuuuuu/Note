**第一，Parse 不是賣資料，是賣能力。** 開發者不是來買「網站資料」的。他們是來讓自己的 agent「能做原本做不到的事」。在 Resy 訂位、搜機票——這些都是 agent 原本不會的，裝了 Parse MCP 就會了。

你的等價物不是「賣法規資料」，是「讓任何 AI agent 裝一個 MCP 就瞬間『懂合規』」。Panta 的 agent 裝上你的 MCP，瞬間知道 50 州 surplus lines 規則。Fenrock 的 agent 裝上你的 MCP，瞬間知道各國 AML 門檻。不需要改架構、不需要寫 prompt、不需要自己研究。

**第二，Parse 的共享池效應。** Parse 最厲害的不是單個 API，是「新建的 API 會加入共享池」。一個用戶讓 Parse 為某個網站建了 API，所有用戶都能用。這是一個越用越強的飛輪。

你的等價物：Panta 的 agent 查詢加州 surplus lines 規則時，你回答了。這個結構化知識進入你的知識庫。下一個保險 AI 新創來查同一條規則時，直接命中。但更重要的是——Panta 查了一個你還沒結構化的法規（比如內華達州的），你去結構化它，然後所有人都能用。**每個客戶的查詢都在擴展你的知識覆蓋。** 這就是之前說的數據飛輪，但 Parse 給了一個非常具體的實現範例。

**第三，零整合成本。** Parse 的 MCP 不需要客戶改任何東西。Claude Code 用戶加一行配置就連上了。這是它三週拿到三萬用戶的核心原因。

你之前的架構裡有一個「整合成本」的摩擦——客戶需要改 agent 架構才能用你的 API。但如果你做的是 MCP tool，整合成本變成零。任何用 Claude Code、Cursor、或任何 MCP 相容 agent 的人，加一行就能用。

**第四，也是最重要的：Parse 證明了 MCP 是 AI agent 生態的「App Store 時刻」。**

在 Parse 之前，agent 要跟網站互動需要自己寫爬蟲。Parse MCP 之後，裝一個工具就行了。這跟 iPhone App Store 的邏輯一樣——手機本身做不到的事，裝一個 app 就會了。

MCP 就是 AI agent 的 App Store。Parse 是其中一個 app（「讓 agent 能操作網站」）。你可以是另一個 app（「讓 agent 懂法規」）。

**這重新定義了你的產品形態。**

之前你想的是：我建一個合規知識資料庫 → 開放 API → 客戶的 agent 來查詢。

現在更好的想法是：我建一個 **Compliance MCP Tool** → 任何 agent 安裝即用 → 每次查詢都擴展知識庫 → 飛輪轉起來。

差異在哪？API 是客戶主動來接的（需要他們知道你存在、做整合決策、改架構）。MCP tool 是客戶在 agent 的工具市場裡「裝一下」就行的（就像裝一個 Chrome extension）。分發成本完全不同。

這也改變了你的 go-to-market。你不需要一家一家去說服 Panta、Fenrock 來接你的 API。你把 Compliance MCP 放到 MCP 工具市場裡，任何用 Claude Code 做合規相關工作的開發者自己就會找到你。就像 Parse 不用一個一個去找客戶——他們把工具放出去，三週三萬用戶自己來了。

## 核心洞察：你不是在建「資料庫」，你是在建「監管理解引擎」

Parse 的啟發其實指向一個更深的東西。Parse 不是一個「網站資料庫」——它是一個**能力引擎**，能讓 agent 即時獲得跟任何網站互動的能力。底層是一個自我擴展的系統：每個用戶建的 API 都進入共享池，系統越用越強。

你看到的是同樣的模式可以在監管領域重現，但更強大。因為監管知識比網站互動有一個根本差異：**監管知識需要持續維護。** 網站的 API 建好了可能半年不變。但法規每天都在改。這意味著你的系統不只是「越用越廣」（覆蓋更多法規），還是「越用越準」（每次法規變更都被追蹤和更新）和「越用越深」（每次真實案例的結果都回流到知識裡）。

用一個具體場景來說明這個系統的威力：

一家台灣的 fintech 想進入印尼市場。他們的 AI agent 通過 MCP 問你的系統：「在印尼提供 P2P lending 服務需要什麼執照？」

如果你的知識庫已經有印尼 OJK 的 P2P lending 法規，系統直接回答——結構化的義務、要求、罰則、申請流程、最近一次修改日期。

如果沒有，系統做三件事：標記這是一個未覆蓋的需求，自動去 OJK 網站（或通過 Parsagon/Parse 等上游工具）抓取相關法規原文，用 LLM 做初步結構化，標記 confidence 為「未驗證」，回傳給 agent 同時提醒這是初步結果。

然後——這是關鍵——當這家 fintech 真的去申請了，結果回流：申請成功了嗎？OJK 要求補充什麼文件？哪些要求是法規裡寫了但實際上不執行的？哪些是法規裡沒寫但實際上必須做的？

**這些結果數據是任何爬蟲工具都拿不到的。** Parse 能爬到 OJK 網站上的法規文件。Parsagon 能追蹤法規變更。但「這條法規在實際執行中是怎麼被應用的」——這個知識只能從你的 Autopilot 服務或客戶 agent 的真實結果中獲得。

這就是你之前分析的 Eve 三環飛輪：查詢量 → 準確度改善 → 執行結果數據。第三環是不可複製的護城河。

## 這個系統能服務的不只是 AI agent

你提到了「不同種類的產品型態」。這是對的。同一個自我演化的監管理解引擎，可以同時以所有七種產品型態交付：

**作為 MCP Tool（給 AI agents）：** Panta 的 agent 安裝你的 MCP，即時獲得 surplus lines 合規能力。Fenrock 的 agent 安裝同一個 MCP，即時獲得跨境 AML 能力。不同的 agent 用同一個工具，但查詢不同的法規，每次查詢都在擴展知識庫。

**作為 Copilot（給人類合規官）：** 一家銀行的合規主管問：「澳洲 2026 年 3 月的 AML/CTF 新規具體改了什麼？對我們的客戶盡職調查流程有什麼影響？」系統回答的不是法規全文，是結構化的影響分析——哪些內部流程需要改、改什麼、截止日期是什麼。

**作為 Autopilot 服務（你自己下場做）：** 你幫客戶直接處理具體的合規任務。每次處理都在餵養系統。

**作為 Platform（給合規團隊）：** 企業合規團隊在上面管理他們關注的所有法規變更、追蹤合規狀態、產生報告。Complyance 模式。

**作為 Agent（端到端）：** 系統不只回答問題，直接幫你完成合規流程——填申請表、提交文件、追蹤審批進度。

**作為 Translation Layer（給現有系統）：** 坐在企業的 ERP/CRM 和監管機構之間，自動翻譯法規變更為系統配置更新。

**作為 Knowledge Hub（開源層）：** Schema 開源，結構化的法規知識開放查詢，建立標準。

關鍵是：**底層是同一個引擎。** 知識的獲取、結構化、驗證、更新——這一層只建一次。但交付層可以是任何形態，取決於客戶是誰、怎麼消費。

## 這個系統跟 Parsagon 的根本差異

Parsagon 做的是：**自動化爬蟲管線 → 原始文件。** 他們的 AI 解決的是「怎麼從政府網站拿到文件」的問題。厲害，但產出是文件。

你的系統做的是：**原始文件 → 結構化知識 → 業務語境推理 → 結果回流 → 知識改善。** 你的 AI 解決的是「這些文件對這個具體業務情境意味著什麼」的問題。產出是判斷。

Parsagon 的數據流是單向的：政府 → Parsagon → 客戶。

你的數據流是雙向的：政府 → 你 → 客戶 → 結果回流 → 你（變得更聰明）→ 所有客戶都受益。

這就是為什麼你說「直接競爭也沒關係」是對的。就算 Parsagon 也提供加州 surplus lines 的法規文件 API，你提供的是「加州 surplus lines 對你這個具體保單的合規要求是什麼，以及歷史上類似保單的通過率是多少」。一個是原料，一個是成品。客戶兩個都需要，或者只需要你的（因為你的已經包含了原料的價值）。

## 這改變了你應該怎麼想這家公司

之前你的思路是自下而上的：選一個利基（surplus lines）→ 結構化一批法規 → 開放 API → 慢慢擴展。

現在可以想的是：**建一個引擎（自動獲取 + 結構化 + 驗證 + 維護法規知識的系統），然後讓這個引擎被任何方式消費。**

起步還是一樣的——你仍然需要一個具體的利基來啟動引擎、產生第一批知識、建立第一個飛輪。但你的架構設計應該從第一天就是「可以擴展到任何監管領域」的，而不是「專門為 surplus lines 建的」。

這就是 Parse 真正的啟發：Alex 沒有建「Resy 訂位 API」和「機票搜尋 API」。他建了一個「把任何網站變成 API 的引擎」。具體的 Resy API 和機票 API 是引擎的產出，不是引擎本身。

你不應該建「surplus lines 合規資料庫」。你應該建「把任何法規變成 agent 可消費的結構化知識的引擎」。surplus lines 是引擎的第一個產出。

但——這是關鍵的「但」——**引擎本身不是你的起步方式。** Parse 也不是第一天就建了通用引擎。Alex 先建了球鞋搶購 bot，然後做網站爬蟲，然後才抽象出通用引擎。

你的路徑同樣是：先用 Autopilot 手動處理一個利基的合規任務 → 在做的過程中發現哪些步驟可以自動化 → 把這些自動化抽象成引擎 → 引擎自然擴展到其他領域。

**引擎是長出來的，不是設計出來的。** 但你現在知道它最終會長成什麼樣——這讓你在架構每一步時都能留好擴展的空間。

想一下 agent 在做任何決策時需要查詢的「規則」有哪些種類：

**法規（Regulation）**——政府規定你必須做什麼、不能做什麼。surplus lines 稅率、AML 門檻、GMP 要求。這是之前一直在討論的。

**行業標準（Standard）**——不是法律但業界必須遵守的。ISO 9001 品質管理、ASHRAE 90.1 建築能效、IEEE 電氣標準。違反不會被罰款，但會失去客戶和認證。

**業務規則（Business Rule）**——企業內部的流程和邏輯。採購審批流程、BOM 結構規則、專案管理的 gate review 條件。不是法律也不是行業標準，但 agent 執行時必須遵守。

**物理/工程約束（Constraint）**——現實世界的限制。鋼材的承重極限、混凝土養護時間、供應鏈的 lead time、製造公差。agent 在規劃時如果不考慮這些，計畫根本不可執行。

**最佳實踐（Best Practice）**——不是必須但應該遵守的。施工排程的常見陷阱、某個供應商的歷史交期表現、某種材料在特定氣候下的表現。agent 要做出「好的」決策而不只是「不違規的」決策，就需要這些。

一個供應鏈 agent 在選供應商時需要同時考慮：這個零件在目的國需要什麼認證（法規）、材料規格要符合什麼 ISO 標準（行業標準）、採購金額在什麼門檻以上要走招標流程（業務規則）、這個供應商的工廠到港口的運輸時間（物理約束）、類似零件歷史上的品質問題分布（最佳實踐）。

**這五種規則的結構其實是一樣的。** 每一條規則都有：適用條件（什麼情境下這條規則生效）、要求（具體要做什麼或不能做什麼）、後果（違反會怎樣）、例外（什麼時候可以不遵守）、來源（誰定的、什麼時候定的）、有效期（什麼時候到期或下次更新）。
