## 逐一拆解每家的壟斷邏輯

### WorkOS：「合規稅」模式

WorkOS 的洞察：每一家 SaaS 公司要賣給大企業，就必須有 SSO、目錄同步、審計日誌。這不是選擇，是企業客戶的硬性要求。自己建這些東西要三個工程師花三個月。用 WorkOS 幾行 code 搞定。

**壟斷來源：** 標準鎖定。一旦你的 SSO 是透過 WorkOS 接的，換掉意味著重新接所有企業客戶的身份提供者。而且 WorkOS 的 free tier 極其慷慨（100 萬 MAU 免費），所以所有新創都從 WorkOS 開始，長大後付費。

**弱點：** 在價值鏈的較高位置。AWS Cognito、Google、Microsoft 隨時可以做同樣的事。護城河主要是開發者體驗和切換成本，不是不可複製的資產。

### Plaid：「翻譯層」模式

Plaid 的洞察：每家銀行的 API 格式不同。Fintech 要連接 12,000 家銀行，不可能自己一家一家接。Plaid 做統一的翻譯層。

**壟斷來源：** 連接數量。接了 12,000+ 銀行，任何新進者要從零開始。而且 Plaid 兩邊都有網絡效應——銀行越多 fintech 越想用，fintech 越多銀行越願意接。

**弱點：** Hockey 親身體會的——Plaid 不控制銀行的基礎設施。銀行的 API 爛，Plaid 也沒辦法。而且銀行可以自己開好 API 繞過 Plaid。這就是為什麼 Hockey 離開去做 Column。

### Checkr：「數據飛輪」模式

Checkr 的洞察：每做一次背景調查，Checkr 的數據庫就多一筆記錄。十年後，Checkr 見過的人比任何競爭對手都多，所以查詢速度更快、結果更準。

**壟斷來源：** 專有數據的累積。100,000 客戶、每年 3000 萬+ 次調查產生的數據是不可複製的。新進者第一天的數據庫是空的。而且 Checkr 定義了背景調查 API 的標準格式——Uber、Lyft 的系統都按 Checkr 的回應格式設計。

**弱點：** 相對較窄的產品（主要是背景調查）。不過 Checkr 正在擴展到「人員基礎設施」。

### Column：「成為底層」模式

Column 的洞察：所有 fintech 基礎設施（Plaid、Stripe、Checkr）最終都依賴銀行。但銀行的技術是 1970 年代的。與其在舊基礎上建抽象層，不如從零建一家新銀行。

**壟斷來源：** 國家銀行牌照 + 自建技術堆疊 + 直連聯準會。沒有中間層。不可繞過。牌照本身就是壟斷——國家銀行牌照不是想拿就拿的。

**弱點：** 需要極大的前期投入（買銀行、建技術、取得監管信任）。成長較慢。但一旦建成，護城河最深。

## 對你的四種路徑對映


| 模式         | 在合規領域的等價物                                      | 你的版本                                                 |
| ---------- | ---------------------------------------------- | ---------------------------------------------------- |
| **WorkOS** | 讓 AI 應用「合規可用」的 API 層——加幾行 code 就讓你的 agent 能查法規 | 合規準備度 API：AI agent 整合一次就能查詢法規、記錄合規決策、產生審計日誌          |
| **Plaid**  | 連接 AI agent 和法規數據源的翻譯層——統一不同國家、不同格式的法規         | 法規連接 API：不管法規來自美國聯邦、加州、還是台灣金管會，你的 API 回傳統一格式         |
| **Checkr** | 合規查詢的執行層——AI agent 問一個問題，你回傳可執行的答案             | 合規查詢 API：「這個人可不可以在德州做保險經紀？」→ 回傳 yes/no + 具體要求 + 來源引用 |
| **Column** | 成為合規知識的「ground truth」本身——從法規原文開始，自己結構化、自己驗證    | 合規知識銀行：擁有結構化法規數據的權威來源，所有 AI agent 建在你上面              |


## 哪個模式最適合你？

**不是四選一。是四層的演化路徑。**

Column 現在同時提供四層的東西：管道（底層）、帳戶（中層）、Card Programs（上層）、Lending（最上層）。但它從最底層開始建。

你的演化路徑：

**起點：Checkr 模式（執行層）** → 幫客戶做具體的合規查詢。「這個州的 surplus lines 稅率是多少？」你手動查、結構化、回傳答案。不需要 API，用 email 都行。這就是你的 Autopilot 服務。

**第二步：WorkOS 模式（準備度層）** → 把你做過的查詢打包成 SDK。AI agent 加幾行 code 就能調用你的合規知識。Free tier 吸引新創，付費版服務成長中的公司。

**第三步：Plaid 模式（連接層）** → 擴展到多個管轄區、多種法規類型。你的 API 成為統一介面，不管法規來自哪裡，AI agent 都用同一個格式查詢。定義 schema 標準。

**第四步：Column 模式（底層）** → 你不再依賴任何第三方法規數據源。你直連法規原始來源，自己結構化，自己驗證。你就是 ground truth。所有合規 AI 公司建在你上面。

**從 Checkr 開始（做一件具體的事），用 WorkOS 擴張（API 化），用 Plaid 連接（多管轄區），最終成為 Column（底層基礎設施）。**

這四家公司不是四個選擇，是你的四個階段。
