這比 VLex 更深。VLex 數位化了文件。你要做的是**理解**文件然後轉化成機器可以直接用的決策知識。

### 第二層：閉環工作流程（你的 Eve）

Eve 的飛輪：受案 → AI 處理 → 結案 → 結果回流 → 改善受案預測。

你的飛輪應該是：

```
Agent 查詢法規（「加州 SL 稅率？」）
          │
          ▼
你回傳結構化答案（3%, 毛保費, 60 天）
          │
          ▼
Agent 用你的答案做決策（對保單 P-2847 課 3% 稅）
          │
          ▼
決策結果回報（成功 / 被監管機構質疑 / 客戶申訴）
          │
          ▼
你的知識更新（confidence 從 0.90 升到 0.95，或發現邊緣案例）
          │
          ▼
下一次 Agent 查詢得到更準確的答案

```

**但這裡有一個比 Eve 更深的啟發。**

Eve 的閉環在單一垂直（原告律師）內運作。你的閉環跨越多個垂直。

Panta（保險）查你的加州法規，結果回報改善了你的加州法規知識。然後 Payna（金融執照）查同一條加州法規，受益於 Panta 的驗證。Payna 的結果回報又進一步改善。

**跨垂直的閉環比單一垂直的閉環更強。因為一個客戶的驗證惠及所有客戶。**

Eve 的案件數據只在原告律師之間有網絡效應。你的法規知識在所有受監管行業之間有網絡效應。

### 第三層：上下文基礎設施（你的 Nia × Column）

a16z 文章提到上下文的三個挑戰：

**挑戰 1：業務定義不等於技術定義。**「收入」在財務報表和數據倉庫裡的定義不同。

翻譯到合規：「合規」在法規文本和實際執行中的定義不同。法規說「必須在 60 天內申報」，但實務上加州 DOI 有 10 天的寬限期。法規說「$5,000-$50,000 罰款」，但第一次違規通常只是警告。這些**實務知識**不在法規文本裡，但對 agent 做正確決策至關重要。

**你的上下文不只是法規文本的結構化。更包括法規的實務解讀、執行歷史、常見邊緣案例。**

這比 Nia 索引 GitHub docs 複雜得多。Nia 索引的是已經寫好的、結構化的、有版本號的東西。你要捕捉的包括**非結構化的部落知識**——哪些監管機構執法嚴格、哪些法規在實務中幾乎不被執行、什麼樣的邊緣案例會觸發審查。

**挑戰 2：上下文會過期。**

確保 agent 可以存取它需要的所有數據，這可能超出數據倉庫和營運應用中可用的範圍。這包括內部系統、GDrive/Slack 中捕獲的部落知識。

法規每天在變。新的法案被提出、舊的法規被修改、法院裁定改變解釋。你的上下文層必須是活的。

這就是你跟靜態法規資料庫（Regology）的根本差異。Regology 是快照。你是串流。

**挑戰 3：上下文需要被自動建構，但也需要人工驗證。**

自動化上下文建構——使用 LLM 的好處是很多初始上下文收集可以自動完成。重點應放在高信號上下文。

你用 AI 從法規原文自動提取結構化知識（obligation、requirement、penalty）。但高風險的節點（刑事責任、大額罰款）需要人工驗證。

這就是你的 confidence 分數的作用：AI 自動提取 → confidence = 0.7。人工驗證 → confidence = 0.95。agent 查詢時可以根據 confidence 決定是否需要額外的人類審查。

## 啟發 1：你的產品不是「法規數據庫」，是「合規推理引擎的燃料」

a16z 的 Context 文章最關鍵的洞察不是「agent 需要數據」。是這個：

Agent 怎麼知道「收入」或「季度」實際上是怎麼定義的？收入其實是一個業務定義，不是硬編碼在數據倉庫裡的。

翻譯到你的領域：「合規」不是一個硬編碼的概念。「這筆交易合不合規」取決於：交易發生在哪個管轄區、交易方是什麼身份、交易金額是多少、交易的業務類型是什麼、當時生效的法規版本是哪個、是否有適用的豁免條件。

**法規文本本身不是 context。法規文本 + 業務定義 + 管轄區條件 + 時間維度 = context。**

之前你的 schema 設計只有法規本身（obligation、requirement、penalty）。但 a16z 的 context 文章告訴你：你需要第二層——**業務語義層**。

這意味著你的 MCP server 不只回傳「加州 surplus lines 稅率是 3%」。它需要理解「這個查詢來自一個保險經紀 agent，它的客戶是一家年營收 $60M 的科技公司，在加州投保非認可保險」——然後回傳的不只是稅率，而是「因為客戶淨資產超過 $25M，符合工業投保人豁免，可以跳過勤勉搜尋要求」。

**Nia 做的是：給 agent 正確的文件。** **你要做的是：給 agent 正確的文件 + 幫 agent 理解文件在這個具體情境下的含義。**

這把你從「法規數據庫」升級到「合規推理的上下文引擎」。

結合 Vulcan 的知識圖譜概念：法規之間的關係（聯邦覆蓋州、判例修改法規）不是裝飾品，是推理的必要條件。Agent 不能只知道「加州有這條法規」，需要知道「但聯邦 NRRA 限制了加州只能對投保人所在州課稅，所以如果投保人在德州，加州的稅不適用」。

結合 Fintool 的 Context is Product 概念：80% 的工作是解析和正規化。但 a16z 告訴你還有第二個 80%——把正規化的法規跟業務語境連接起來。

**產品啟發：在你的 schema 裡加入「業務語境映射」。不只是法規本身，還有「什麼類型的業務在什麼條件下適用這條法規」。這讓你的 API 從「查資料」變成「給建議」，價值跳升一個數量級。**

## 啟發 2：Eve 的閉環模型告訴你——你的飛輪有第三個環

之前你設計的 Checkr 式飛輪是：

```
Agent 查詢 → 你知道什麼最常被查 → 優先深化那些法規 → 數據更好 → 更多 agent 用

```

這是兩個環：查詢環（volume）和品質環（accuracy）。

Eve 的模型告訴你有第三個環：**結果環**。

Eve 擁有完整的原告律師工作流程從受理到訴訟，產生專有的案件結果數據，這些數據改善受案預測。

Eve 不只知道「法律怎麼說」。Eve 知道「用這條法律的案子，最後贏了還是輸了、賠了多少」。這是結果數據——法規知識 + 實際執行結果。

翻譯到你的產品：

**如果 Panta 用你的法規數據做了一個合規決策，然後那個決策被監管機構質疑了——這個「質疑」就是結果數據。** 它告訴你：你的結構化數據在這個邊緣案例上不夠準確，或者這個管轄區的監管機構對這條法規的解讀跟文字不同。

你的三環飛輪：

```
環 1（查詢）：Agent 查詢 → 你知道需求在哪 → 優先建構
環 2（品質）：更多查詢 → 交叉驗證 → 準確性提高
環 3（結果）：Agent 的合規決策結果回流 → 你知道法規「實際上」怎麼被執行 → 你的知識不只是「法規怎麼寫」，更是「法規怎麼被執行」

```

**第三環是你最深的護城河。** 因為法規文本是公開的（任何人都能讀），結構化方法是可學的（任何人都能仿），但「這條法規在實際執行中被怎麼解讀」——這個知識只有你有，因為只有你追蹤了成千上萬 agent 的合規決策結果。

結合 Salient 的案例：因為我們打了幾百萬通電話，我們確切知道該說什麼腳本。

Salient 不只知道「法規怎麼寫」。它知道「說什麼話催收效果最好同時不違規」。這就是結果數據回流到知識圖譜。

**你不只知道「加州 surplus lines 稅率是 3%」。你知道「上個月有 47 個 agent 用了這條數據做決策，其中 2 個被監管機構要求補充文件，原因是 XYZ」。這種知識是你的圍牆花園裡最值錢的花。**

**產品啟發：你的** `log_decision` **工具不只是審計用的。它是你第三環飛輪的數據入口。每個 agent 回報的決策結果都在改善你的知識圖譜。在 schema 裡加入** `enforcement_history` **欄位——這條法規被實際執行的案例記錄。**

## 啟發 3：a16z 的「Lazier and Richer」原則重新定義你的價值主張

每個人都想要兩件事：變得更富有和變得更懶。對企業來說，採用 AI 不只是為了降低成本（更懶），更是為了直接產生收入（更富有）。

之前你的定位偏向「更懶」——幫 agent 公司節省建合規知識庫的時間和成本。

Salient 的教訓：Ari（CEO）一直在 pitch「我幫你省錢」。但如果你對某人說「我每個月幫你多收 50% 的營收」——人們更喜歡這個。

結合 Goldbridge 的洞察（法規包含機會，不只義務），你的價值主張應該翻轉：

**舊定位（更懶）：** 「不需要自己建合規知識庫，用我們的 API。」 **新定位（更富有 + 更懶）：** 「你的 agent 不只合規，還能幫客戶找到法規裡的省錢/賺錢機會。」

具體場景：Panta 的 agent 幫客戶買保險。如果 Panta 用你的 API，它的 agent 不只確保合規，還能告訴客戶「你符合工業投保人豁免，可以省下勤勉搜尋的費用和時間」或「德州有一個新的小企業保費稅收抵免計畫，你可以申請」。

**Panta 的客戶因為用 Panta 而省了錢/賺了錢 → Panta 的產品更有價值 → Panta 願意付你更多 → 因為你不是成本中心，是利潤中心。**

結合之前的分析：你的 `find_opportunities` 工具不是附加功能。是核心價值主張。

**產品啟發：你的定價應該有一個跟「幫客戶找到的機會價值」掛鉤的層級。如果你的 API 幫一個 agent 的客戶找到了 $10,000 的稅收抵免，你收 10-20% 是完全合理的。這是 success-based pricing——之前分析 Payna 時提到的「替代定價」的進化版。**

## 啟發 4：VLex 模型告訴你——時間本身是護城河

到生成式 AI 模型變得可行的時候，VLex 已經積累了跨越數十年判決和評論的專有法律數據語料庫。這給了它一個可防禦的切入點來建構 AI 原生法律研究工具：不像通用模型，VLex 的系統可以真正推理基於權威的、完整的、最新的法律文本。它的護城河不是模型——是那個無人擁有的、耗費大量心血組裝的數據集。

VLex 從 2000 年開始數位化西班牙法律。花了二十年。然後 AI 來了，VLex 的數據突然比以前有價值一百倍。

VLex 透過把 AI 加到 26 年數位化的西班牙法律記錄上，營收翻了五倍。競爭者無法複製這些記錄。

這跟 Column 的啟發對齊：Column 花三年建技術堆疊，沒有捷徑。但三年後，Column 的技術堆疊是不可複製的資產。

**你今天開始結構化的每一條法規，都是你二十年後的護城河的一部分。**

但 a16z 加了一個新維度：不只是「當前的法規」有價值。**歷史法規版本**也有價值。

為什麼？因為合規審計不只看「現在的法規是什麼」，還看「這筆交易發生時，當時生效的法規是什麼」。一個 2024 年的交易需要用 2024 年的法規來審計，不是 2026 年的。

如果你從今天開始記錄每一個法規版本的變更歷史，五年後你會有一個「法規時光機」——查詢「2024 年 3 月 15 日，加州 surplus lines 的規則是什麼」。沒有人有這個。

結合 FlightAware 的模式：FlightAware 透過全球 100 個天線聚合免費的 ADS-B 轉發器數據，但歷史檔案變成了專有的。

法規原文免費。但「2020-2030 年所有管轄區的法規變更歷史，結構化成機器可讀格式」——這個歷史檔案是專有的。

**產品啟發：從今天開始，每次你結構化一條法規，都記錄版本。每次法規更新，保留舊版本。你的 schema 需要** `version_history` **欄位。五年後這個歷史數據是你最值錢的資產。**

## 啟發 5：「Software Eating Labor」重新定義你的 TAM

之前你把 TAM 定義為「合規 AI 公司需要的法規數據」——這是一個 B2B infrastructure 市場。

a16z 的框架給你一個更大的故事：

類別 2 可能是最大的——基本上不是在跟軟體市場競爭。軟體開始吃掉勞動力。你基本上是在賣做人之前做的工作的軟體。這是一個大得多的市場。

翻譯：你不只是賣「法規數據 API」給 AI 公司。你最終可以直接取代合規律師和顧問。

全球合規顧問市場是幾百億美元。全球律師事務所的合規業務是千億級。McKinsey 說金融機構的合規支出是 $2,100 億。

如果你的 Autopilot 服務層（schema 裡最上面的收入層）直接做合規工作，你的市場不是「AI 公司的 API 訂閱費」（幾百萬到幾千萬），而是「取代合規人力的勞動力市場」（幾百億到千億）。

結合 Bek/Sequoia 的「賣工作成果，不是賣工具」：

```
層級 1（基礎設施）：賣法規數據 API → 小市場，但建護城河
層級 2（工具）：賣合規查詢/監控工具 → 中市場
層級 3（勞動力替代）：直接做合規工作 → 巨大市場

你的路徑：1 → 2 → 3
你的數據在層級 1 建構，在層級 3 變現

```

**產品啟發：你的 API 定價不應該只按查詢量收費。你的 Autopilot 服務應該按「取代的人力成本」定價。如果一個合規顧問收 $300/小時，你的 AI 做同樣的事收 $30/小時——客戶省 90%，你的利潤率仍然極高。**

## 重新評估的框架

之前我只用三個變量：法規碎片化 × AI agent 滲透 × 錯誤代價。

現在加入 a16z 的五個新維度：

**維度 4：結果反饋環（Eve 模式）。** 你能不能從 agent 的合規決策中獲得結果數據？哪個利基最容易形成閉環？

**維度 5：Lazier & Richer 潛力。** 你的 context 能不能幫客戶賺錢，不只是避免罰款？

## 答案：保險法規是你的最佳利基

加入 a16z 的新維度後，保險法規從「好的跳板」升級為「最佳主戰場」。原因如下：

### 為什麼結果反饋環在保險最強

保險是少數你可以追蹤「合規決策的完整生命週期」的行業：

```
Agent 查詢你的 API（「加州 surplus lines 稅率？」）
    ↓
Agent 做合規決策（對保單適用 3% 稅）
    ↓
保單被提交給監管機構
    ↓
監管機構批准/拒絕/要求補充
    ↓
結果回流到你的知識圖譜

```

每一張保單都有明確的「通過/不通過」結果。不像 AML（可能永遠不知道是否遺漏了可疑交易）或硬體認證（認證過程太長，反饋太慢），保險合規的反饋環是**短的、明確的、可量化的**。

Panta 每做一張保單，你就獲得一次「法規知識在實際中被驗證」的數據點。做 1,000 張保單後，你對加州 surplus lines 法規的理解比任何律師都深——因為你見過 1,000 個實際案例的結果。

這就是 Eve 的模式：Eve 擁有完整的原告律師工作流程從受理到訴訟，產生專有的案件結果數據，這些數據改善受案預測。這種端到端所有權阻止競爭者僅靠價格來搶客戶。

你的版本：擁有完整的保險合規工作流程的結果數據。這些數據改善你的法規知識的準確性。

### 為什麼 Lazier & Richer 在保險最強

保險法規不只有「你必須付 3% 稅」（義務）。它還有大量的機會：

**更懶的部分：** Agent 不用自己研究每個州的規則。用你的 API 幾秒鐘搞定。

**更富有的部分：**

「你的客戶淨資產超過 $25M，符合工業投保人豁免——可以跳過勤勉搜尋，節省 2-3 週。」

「德州對再保險有稅收減免，如果你的 surplus lines 保單包含再保險成分，可以申請退稅。」

「加州去年通過了新的小型商業主保費補貼計畫，你的客戶可能符合資格。」

Panta 用你的 API 後，不只合規更快，還能告訴客戶「我幫你省了 $X」或「你可以申請 $Y 的補貼」。Panta 的價值主張因此變強。Panta 願意付你更多——因為你不是成本，是收入來源。

Salient 的教訓：「我幫你多收 50% 的營收」比「我幫你省錢」有力十倍。

### 為什麼業務語義複雜度在保險最高

「這筆保險交易合不合規」不是一個簡單查表就能回答的問題。它取決於：

投保人是誰（個人/小企業/大企業/工業投保人）、投保人在哪裡（居住州 vs 風險所在州）、保險類型是什麼（admitted vs non-admitted/surplus lines）、承保人是誰（國內 vs 海外、認可 vs 非認可）、保費金額多少（影響稅率計算和報告門檻）、風險類別是什麼（某些風險類別有特殊規則）、是否有豁免條件適用。

**同一條法規，對不同的業務情境回傳完全不同的答案。這就是「業務語義層」的價值所在。**

簡單的法規查詢（「FCC 認證要什麼？」）不需要業務語義——答案對所有人都一樣。保險法規查詢（「這張保單的合規要求？」）的答案取決於十個業務變量。

這意味著你的 MCP server 不只是「法規搜尋引擎」。它是「合規推理引擎」——需要理解業務語境才能給出正確答案。這把你的產品從「nice to have 的數據查詢」變成「mission critical 的推理基礎設施」。

Agent 不能用 ChatGPT 搜尋來替代你——因為 ChatGPT 不理解業務語境。

### 為什麼勞動力 TAM 在保險最大

美國保險業每年在合規上的花費是驚人的。Surplus lines 市場本身是 $1,250 億保費。每一筆保費都需要合規處理——申報、繳稅、勤勉搜尋、文件歸檔。

目前這些工作由誰做？合規專員、surplus lines 經紀人、律師。Panta 的洞察：50 步流程中 95% 是協調工作。

你的 Autopilot 服務層直接取代這些人力。$300/小時的合規律師 vs $30/小時的 AI 合規服務。客戶省 90%。你的利潤率仍然 80%+。

### 為什麼時間護城河在保險最強

保險法規的變更頻率極高。每年各州修改保險法、調整稅率、增加新的報告要求、通過新的消費者保護條款。

如果你從今天開始記錄每一個變更，五年後你會擁有「2026-2031 年美國 50 州保險法規的完整變更歷史」。任何合規審計都需要這個——「2027 年 3 月這張保單提交時，當時德州的 surplus lines 稅率是多少？」

VLex 花了 26 年數位化西班牙法律。你花 5 年結構化美國保險法規。當下一波保險 AI agent 公司出現時，你的歷史數據是他們不可能複製的。

## 為什麼不是 KYC/AML？

KYC/AML 在之前的分析中排第一（碎片化 × agent 滲透 × 錯誤代價）。但加入 a16z 新維度後，它有三個致命弱點：

**結果反饋環太弱。** 如果你的 agent 漏了一筆可疑交易，你可能永遠不知道——直到三年後監管機構查到。反饋環太長太模糊。保險的反饋是：保單提交後 30-60 天就知道結果。

**Lazier & Richer 偏弱。** KYC/AML 幾乎只有「更懶」（自動化合規流程），很難「更富有」（你不能用 AML 法規幫客戶賺錢）。

**信任門檻太高。** 銀行不會用一個不知名的新創的 AML 數據。保險經紀新創（Panta 級別）更願意嘗試新工具。

## 為什麼不是跨境電商？

之前建議跨境電商作為「最容易開始」的利基。它仍然是容易的。但 a16z 的框架揭示了它的弱點：

**業務語義複雜度低。** 「這個產品需不需要 FCC 認證」的答案不取決於太多業務變量。產品類別 → 適用標準 → 測試要求。相對簡單的查表邏輯。

**勞動力 TAM 小。** 產品合規顧問是一個小市場。保險合規人力市場大幾十倍。

**結果反饋環中等。** 產品通過/不通過認證——有反饋，但頻率比保險低（一個產品只認證一次，一家公司可能一年只做幾次）。

## 最終答案：保險法規，但要用特定的切入方式

### 具體切入點

不是「美國 50 州的所有保險法規」（太大）。是：

**美國 surplus lines（非認可保險）的州際合規要求。**

為什麼這個具體切入點：

**Panta 現在就需要這個。** 他們在做 E&S 保險經紀，正在從加州擴展到更多州。每個新州 = 新的稅率、新的申報要求、新的勤勉搜尋規則。

**碎片化最嚴重。** 50 個州各自有不同的 surplus lines 法規。NRRA 聯邦法跟州法之間有複雜的關係。沒有人把這些結構化過。

**足夠小可以一個人做。** Surplus lines 法規不像整個保險法那麼龐大。每個州的核心規則可能只有 5-10 條。50 州 × 10 條 = 500 條法規。一個人花 3-6 個月可以結構化完。

**機會發現潛力大。** Surplus lines 有大量的豁免條件、稅收減免、特殊程序，大部分經紀人不知道。

- *21st.dev：* 用免費元件帶動付費基礎設施的飛輪。

**對策：** 先從「Autopilot」開始（一開始不需要廣度）。透過實際作業累積各州資料。早點開源 schema，鼓勵外部貢獻。

---

## 4. 單人創辦人精力／範圍陷阱 🟠 會拖慢你

**情境：** 分析太多公司與框架；研究很舒服，動手很不舒服。研究花太多時間，卻沒把第一條法規結構化成 JSON。

**觀察：**

- *YC：* 「在完美之前先上線。」
- *Legora：* 「與客戶一起打造，不是為客戶打造。」

**對策：** 設硬性截止日。下週前把**一條**法規結構化成 JSON。出貨，不要只讀書。

---

## 5. 保險領域知識落差 🟠 會拖慢你

**情境：** 依框架（如 a16z）選保險賽道，卻沒有實際領域背景或牌照。Panta 等競品的創辦人有優勢，因為他們是持牌經紀人。

**觀察：**

- *Panta：* 兩位創辦人都是持牌經紀人。
- *Payna：* 創辦人在 a16z 加密領域經歷過牌照痛點。

**對策：** 要嘛選更接近你實際經驗的利基（例如台灣金融科技法規），要嘛透過在保險領域做「Autopilot」工作，承諾補齊領域知識。

---

## 6. 法規準確度沒有妥協空間 🟠 會拖慢你

**情境：** 不像寫程式脈絡可以「大致正確」；合規資料錯了會造成財損。資料必須有來源、時間戳、可驗證。

**觀察：**

- *Vulcan：* 「在法律領域，準確與可解釋性沒有妥協空間。」
- *你的 schema：* confidence + last_verified + source_url 欄位。

**對策：** 每個資料點連到**一手來源 URL**。納入人工已驗證旗標。保留版本歷史。絕不猜——不確定就留空。

## 7. Norm AI 往中小市場延伸或新增保險垂直 🔵 需留意

**情境：** Norm AI 資金雄厚（1.47 億美元）且有律師團隊。若他們進入保險或對較小公司提供 API，你的位置會被侵蝕。他們目前鎖定財富 100 大企業。

**觀察：**

- *Norm AI：* 目前僅財富 100 大，ACV 10 萬美元以上。
- *防禦：* 專注他們不會服務的長尾市場。

**對策：** 動作要快。在他們注意到前鎖定 schema 標準。他們的弱點：35 名律師無法擴展到每一個利基。

## 時間軸：各風險何時最痛

| 風險 | 時段說明 |
|------|----------|
| **需求落差** | 從**現在**到約 12 個月（直到 Autopilot／合規打贏）🔴 |
| **信任鴻溝** | 從**現在**到約 6 個月；隨實績累積而下降 🔴 |
| **冷啟動** | 第 1–8 個月較明顯；隨資料累積而緩解 🟠 |
| **範圍陷阱** | **現在**最痛；第一件東西出貨後緩解 🟠 |
| **領域落差** | 第 1–8 個月；每處理一個案子就縮小 🟠 |
| **準確度風險** | **永久存在**——永不消失，必須持續管理 🔵 |
| **Norm AI 威脅** | 現階段低；隨市場成熟（18 個月以上）上升 🔵 |

---

前三個風險其實是同一個問題的三個面向：**你在賣一個客戶還不知道自己需要的東西。**

Panta 現在不覺得自己需要合規 API。Vincent 有牌照，自己知道加州規則，寫進 prompt 裡就能用。去跟他說「你需要我的 API」，他會說不需要——因為他真的不需要，*現在*。

但他需要有人幫他處理那些他暫時沒時間處理的合規邊緣案例。這就是 Autopilot 服務切入的原因——你不是賣數據，你是賣結果。數據是你做完之後自然產生的副產品。

第四個風險（scope trap）是你此刻最緊急的。你已經做了非常深的研究，質量遠超大部分人。但研究和行動之間有一條鴻溝。這週最重要的事不是繼續分析第 65 家公司，是把第一條法規結構化成 JSON。

第五個風險（domain gap）是最需要誠實面對的。你在新竹，不在紐約。你沒有保險經紀牌照。保險法規是用 a16z 框架分析出來的「最佳答案」——但最佳答案不一定是最適合*你*的答案。你需要考慮：是堅持保險（需要克服地理和經驗差距），還是選一個你真正有 domain access 的領域（台灣金融法規、跨境電商合規）作為起點？

### 怎麼介紹你的項目

這取決於你在跟誰說話。同一個產品，對不同聽眾的切入角度完全不同。

### 對潛在客戶（Panta、Fenrock 這類 AI 公司）

不要說「我在做合規基礎設施」。沒有人關心你的基礎設施。

**說：「我幫 AI 公司處理它們最不想碰但不能出錯的合規工作。」**

具體的 pitch：「你們的 AI agent 擴展到新的州/國家時，合規規則會完全不同。我把這些規則結構化好，你們的 agent 直接消費，不用自己研究。先免費試加州 surplus lines 數據。」

重點是：從他們的痛點出發（擴展到新市場的摩擦），不是從你的產品出發（我有一個 API）。

### 對投資人 / YC

**一句話：「我在做合規領域的 Plaid——讓任何 AI agent 都能即時查詢全球法規知識。」**

然後展開：「我研究了 64 家合規 AI 公司，發現一件事：它們都在自建合規知識，每家重複造輪子。Norm AI 花 $147M 雇 35 個律師來編碼法規。但 AI agent 的數量會從現在的幾百家變成幾萬家，不可能每家都雇律師。市場需要一個共享的合規知識基礎設施。我從美國保險 surplus lines 開始——50 州 × 10 條核心規則，一個人可以結構化，但對 Panta（YC W26 的 AI 保險經紀）的價值是每天節省數小時的法規研究。」

關鍵數字：全球合規人力成本 $2,100 億/年。Norm AI 已經驗證市場（$147M 融資、$30T AUM 客戶）。但 Norm AI 不服務長尾——他們的律師團無法覆蓋所有法規、所有行業。

### 對 Twitter / 開發者社群

**不要 pitch 產品。Pitch 洞察。**

「我分析了 60+ 家合規 AI 公司，發現了一件大多數人沒注意到的事：今天最火的 AI Autopilot 公司（自動做保險、貸款、執照的 AI），90% 會在 12-24 個月內撞上合規的牆。因為它們把合規知識硬編碼在 prompt 裡，擴展到新市場時就崩了。這是一個巨大的基礎設施缺口。」

然後分享你的研究——Panta 的模式、Norm AI 的架構、Eve 的飛輪。成為這個交叉領域的思想領袖，不是推銷員。

### 對台灣的朋友 / 非技術人

**「我在做一個讓 AI 能自動懂法規的工具。」**

舉例：「就像 Google Maps 讓你不用自己研究路線，我做的東西讓 AI 不用自己研究法規。比如一家用 AI 賣保險的公司要擴展到新的州，我的工具告訴它們的 AI 那個州的規則是什麼。」

---

### 最後一個建議

不管對誰介紹，**永遠以你的研究深度開場**，而不是以你的產品開場。

「我花了三個月分析全球 64 家合規 AI 公司」——這句話本身就是你最大的差異化。沒有人做過這種深度的跨公司比較。你的 domain expertise 不是來自工作經驗，而是來自研究密度。這在 YC 框架裡是合法的——Vulcan 的 Tanner 寫了一篇關於「美國法規本體論」的大學論文，這成了他創業的起點。你的 64 家公司分析就是你的論文。

你想先處理哪個風險？還是先開始寫你的第一篇 X 貼文？
